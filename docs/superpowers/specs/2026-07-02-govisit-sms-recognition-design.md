# Дизайн: распознавание SMS от govisit в otp-bridge.py

Дата: 2026-07-02
Статус: одобрено пользователем

## Проблема

Сейчас `otp-bridge.py` берёт **первое** число из 4–8 цифр из любой входящей SMS
за последние 180 секунд: `SENDER_CONTAINS = []`, `TEXT_CONTAINS = []`,
`CODE_REGEX = \b(\d{4,8})\b`. Любая посторонняя SMS с числом (банк, доставка),
пришедшая в это окно, может подменить код govisit.

## Исходные данные (подтверждено пользователем)

Реальная SMS от govisit, всегда на иврите:

```
קוד האימות לגוביזיט 0189.קוד זה תקף ל-5 דקות
```

- Отправитель в Messages отображается как `GoVisit`.
- Код — 4 цифры, **может начинаться с нуля, а может и нет**; трактуем его как
  непрозрачную строку из 3–8 цифр, длину жёстко не фиксируем.
- В тексте есть цифра-ловушка: «5» в «ל-5 דקות» (действителен 5 минут).
- Якорная фраза `קוד האימות` («код подтверждения») стоит непосредственно перед
  кодом.

## Решение

### 1. Конфиг (блок НАСТРОЙКИ в otp-bridge.py)

- `SENDER_CONTAINS = ["govisit"]` — код берём только из SMS от GoVisit.
  Существующее сравнение регистронезависимое, поэтому `GoVisit` совпадёт.
- `CODE_TTL = 300` — под заявленный в SMS срок жизни кода «5 минут»
  (было 180).
- `TEXT_CONTAINS` остаётся `[]` — фильтр по отправителю плюс якорная
  регулярка достаточны; текстовый фильтр мешал бы fallback-ветке.

### 2. Извлечение кода — новая функция `extract_code`

```python
# Основная: якорь «קוד האימות» (код подтверждения), затем до 20 нецифровых
# символов («לגוביזיט », двоеточие и т.п.), затем сам код.
ANCHORED_CODE_REGEX = re.compile(r"קוד האימות[^0-9]{0,20}(\d{3,8})")
# Fallback — прежняя общая: отдельно стоящее число 4–8 цифр.
CODE_REGEX = re.compile(r"\b(\d{4,8})\b")

def extract_code(text):
    m = ANCHORED_CODE_REGEX.search(text) or CODE_REGEX.search(text)
    return m.group(1) if m else None
```

В `_poll_once` строка `match = CODE_REGEX.search(text)` заменяется на вызов
`extract_code(text)`. Код везде остаётся строкой — ведущий ноль, если он есть,
сохраняется. Якорная регулярка не фиксирует длину кода (3–8 цифр), поэтому
переживёт смену формата 4 → 6 цифр. Fallback страхует на случай, если govisit
переформулирует SMS и якорь исчезнет.

### 3. Тесты — `test_otp_extraction.py`

Стандартный `unittest`, без зависимостей и без chat.db; проверяются чистые
функции `_extract_text` и `extract_code`:

1. Реальный текст SMS → ровно `"0189"` (ведущий ноль на месте).
2. Цифра-ловушка «5» из «ל-5 דקות» не извлекается.
3. Якорь побеждает fallback: в тексте и код после якоря, и постороннее
   6-значное число → берётся код после якоря.
4. Текст без якоря, но с 6-значным числом → срабатывает fallback.
5. Имитация `attributedBody`: UTF-8-байты SMS в бинарном мусоре →
   `_extract_text` + `extract_code` → `"0189"`.
6. Текст без кода → `None`.

### 4. README

- В разделе про запуск моста: дефолты уже настроены под govisit (отправитель
  GoVisit, ивритский формат SMS).
- В «Если код не извлекается»: если отправитель отображается иначе (например,
  короткий номер вместо `GoVisit`) — поправить `SENDER_CONTAINS`.

## Что не трогаем

HTTP-сервер, launchd-plist, userscript (`govisit-monitor.user.js`) — без
изменений. Диффа кроме описанного не будет.

## Критерии успеха

- `python3 -m unittest test_otp_extraction -v` — все тесты зелёные.
- Из реального текста SMS govisit извлекается именно код (с нулём или без),
  а не «5» и не постороннее число.
- SMS от других отправителей игнорируются.
